 1. 重激活放弃权重继承（防脏数据污染）

 论文描述（第207行）：
 ▎ "Inactive components keep their stored weights, which enables weight inheritance if they are reactivated."

 论文明确支持权重继承，未提及重激活时是否清零 optimizer state。

 代码实现（train_integration.py:214-242）：
 def _reset_optimizer_state_for_component(opt, layer, index):
     # 清零 lora_A[index,:] 和 lora_B[:,index] 对应的 Adam m/v 矩
     _zero_matching_state(layer.lora_A.weight, lambda t: t[index, :].zero_())
     _zero_matching_state(layer.lora_B.weight, lambda t: t[:, index].zero_())
 每次 commit_mutation（Expand/Prune/Reallocate）后都调用此函数。

 分析：
 - 权重本身保留（继承），但 optimizer state（Adam m/v）被清零
 - 这是正确的工程决策：旧 m/v 是基于"该组件活跃时"的历史梯度，重激活后权重可能已被其他组件更新，旧 momentum
 会引入错误偏置
 - 论文未提及此细节，但这是实现正确性的关键

 结论：代码实现更好，论文需补充说明
 ▎ 在 Practical Remarks 节补充一句：重激活时保留权重但重置对应的 optimizer 动量状态，防止 stale momentum
 污染新激活组件的早期更新。

 2. rsLoRA 核心参数化（方差稳定与防学习率爆炸）

 论文描述：
 论文完全未提及 rsLoRA。参数化公式（Eq. 100-104）中无 scaling factor，前向传播写为 $\Delta W_\ell = \sum m_i b_i
 a_i^\top$，无缩放项。

 代码实现（evo_rank_lora.py:86-88, 116）：
 def get_scaling_factor(self, r: int) -> float:
     """rsLoRA 风格的缩放因子: alpha / sqrt(r)"""
     return self.lora_alpha / math.sqrt(max(r, 1))

 # 前向传播
 scaling = self.get_scaling_factor(len(active_indices))
 return out * scaling

 分析：
 - 代码使用 alpha/sqrt(r)（rsLoRA），而论文公式无 scaling
 - rsLoRA 的核心意义：当 r 变化时（EvoRank 的核心特性），标准 alpha/r 会导致 scaling 随 r 变化而剧烈波动（r 从 8→4 时
 scaling 翻倍），引发训练不稳定
 - alpha/sqrt(r) 使方差在 r 变化时保持稳定，是动态秩场景下的必要设计
 - 论文缺少此描述会让读者无法复现

 结论：代码实现更好，论文需补充
 ▎ 在参数化小节补充 scaling factor 公式：$\text{output} = (x A_\ell^\top B_\ell^\top) \cdot
 \frac{\alpha}{\sqrt{r_\ell}}$，并说明 rsLoRA scaling 在动态秩场景下的方差稳定性。

 3. 剪枝分位数排除"冷却期"组件（防门槛扭曲）

 论文描述（Algorithm 2, line 295）：
 τ_prune = Percentile({ŝ_{ℓ,i} : m_{ℓ,i}=1}, p_p)
 论文写的是"所有活跃组件"（m_{ℓ,i}=1），未排除冷却期组件。

 但 Algorithm 2 line 304 写：
 if ŝ_{ℓ,i} < τ_prune and r_ℓ > r_min and component (ℓ,i) is not in cooldown
 即冷却期组件不会被剪枝，但参与了阈值计算。

 代码实现（rank_evolution_controller.py:292-306）：
 # 仅收集活跃且不在冷却期的组件计算分位数
 valid_s_list = []
 for name, layer in self.layers.items():
     active_and_not_cool = layer.active_mask & (self.cooldowns[name] == 0)
     if active_and_not_cool.any():
         valid_s_list.append(self.ema_s[name][active_and_not_cool])

 分析：
 - 代码在计算 τ_prune 时排除了冷却期组件，论文在计算阈值时未排除
 - 工程理由：冷却期组件刚被激活，lora_B 列为零（zero-init）或极小（gradient-init），其 s 值接近
 0，若纳入分位数计算会人为压低 τ_prune，导致其他正常组件被错误剪枝
 - 代码实现更合理，论文描述存在歧义

 结论：代码实现更好，论文需修正
 ▎ Algorithm 2 line 295 改为：$\tau_{\mathrm{prune}} = \mathrm{Percentile}({\hat{s}{\ell,i}: m{\ell,i}=1 \text{ and }
 (\ell,i) \text{ not in cooldown}}, p_p)$

 4. 梯度下界近似（防分布式 OOM）

 论文描述（Eq. 138-139）：
 $$g_\ell = \left|\frac{\partial \mathcal{L}}{\partial \Delta W_\ell}\right|_F$$
 论文直接写完整梯度矩阵的 Frobenius 范数，未说明如何高效计算（$\partial\mathcal{L}/\partial\Delta W$ 是 $d\times k$
 矩阵，DeBERTa 中可达 768×3072）。

 代码实现（evo_rank_lora.py:408-452）：
 # 正则化正规方程替代 SVD 伪逆，避免 pinv 底层 SVD 的瞬间显存峰值
 ata = at.T @ at  # (r, r)
 m = torch.linalg.solve(ata + 1e-6*I, at.T)  # (r, in)
 gf_sq = torch.trace(s_mtx @ t_mtx).clamp(min=0.0)
 return float(torch.sqrt(gf_sq).item())
 这是 $|G|_F$ 的下界近似（投影到 A 的行空间），不构造完整 $d\times k$ 矩阵。

 分析：
 - 完整 $|G|_F$ 需要 $G = \partial\mathcal{L}/\partial\Delta W$，在 DDP 下需 all_reduce 一个 $d\times k$
 张量，显存峰值极高
 - 代码用 $|G A^+|_F$（投影近似）作为下界，只需 $(r\times k)$ 规模运算
 - 这是正确的工程权衡，但论文未说明

 结论：代码实现更好，论文需补充
 ▎ 在 Practical Remarks 或 Implementation Details 节说明：实际计算 $g_\ell$ 时使用投影近似（下界），避免构造完整
 $d\times k$ 梯度矩阵导致的显存峰值。

 5. Zero-Gradient 陷阱与双重保护（防出师未捷身先死）

 论文描述（Algorithm 2, line 282-284）：
 if t ≤ T_warm: continue
 论文只提到 warmup 期间禁止结构演化，未解释原因。

 代码实现（evo_rank_lora.py:29-31 注释 + train_integration.py:268-288）：
 # 警告 (Zero-Gradient Trap): 由于 lora_B 零初始化，训练 Step 0 时
 # grad_A = B^T @ G 必为零。外部 Controller 必须遵循 Warmup（前 10% steps
 # 禁止剪枝/扩张），等 B 充分更新后才能评分。
 代码有明确的双重保护：
 1. warmup 期间 should_evolve=False（train_integration.py:268-270）
 2. _is_initialized 标志防止第一帧 EMA 全零（rank_evolution_controller.py:255-267）

 分析：
 - lora_B 零初始化导致 step 0 时 grad_A = B^T @ G = 0，s 值全零，若此时做剪枝会错误移除所有组件
 - 论文提到了 warmup 机制但未解释 Zero-Gradient 陷阱的根本原因
 - 代码的双重保护（warmup + EMA 初始化直接赋值）是正确的

 结论：代码实现更好，论文需补充原因
 ▎ 在 warmup 描述处补充：由于 lora_B 零初始化，训练初期 grad_A 接近零，组件重要性评分不可靠，warmup
 期间禁止结构演化以避免错误剪枝。

 6. Max-scaling 归一化（防评分系统崩溃）

 论文描述（Eq. 217）：
 $$u_\ell = \alpha \tilde{g}\ell + \beta \widetilde{\bar{s}}\ell$$
 论文写"normalized"但未说明归一化方式。

 代码实现（rank_evolution_controller.py:225-254）：
 g_max = max(raw_g.values())
 s_max = max(raw_bar_s.values())
 eps = 1e-12
 g_t = raw_g[name] / (g_max + eps)   # max-scaling: x/x_max
 s_t = raw_bar_s[name] / (s_max + eps)
 curr_u = self.alpha_u * g_t + self.beta_u * s_t

 分析：
 - 代码使用 max-scaling（$x/x_{\max}$），将所有层的分数归一化到 $[0,1]$
 - 优点：保持零映射到零，不改变相对排序，对量纲无关
 - 若不归一化，g 和 s 的量纲差异（g 是梯度范数，s 是参数范数乘积）会导致 α/β 权重失去意义
 - 论文未说明具体归一化方式，读者无法复现

 结论：代码实现更好，论文需补充
 ▎ 在 Dynamic Thresholding 节明确说明：$\tilde{g}\ell = g\ell / \max_{\ell'} g_{\ell'}$，$\widetilde{\bar{s}}\ell =
 \bar{s}\ell / \max_{\ell'} \bar{s}_{\ell'}$（max-scaling，保持零映射到零）。

 7. Alg 1 与 Alg 2 职责分离（回归 ES 算法本质）

 论文描述：
 - Algorithm 1（主训练）：每 K 步做一次 ES 结构更新，生成 λ 个候选，elitist 选择
 - Algorithm 2（动态阈值）：基于 EMA 统计量和分位数阈值决定候选集

 代码实现（train_integration.py:268-390）：

 - controller.update_statistics() → controller.generate_mutations() → 逐一 apply/eval/undo → commit best（对应 Alg 1 的
  ES 循环）
 - generate_mutations() 内部调用 Alg 2 的阈值逻辑

 分析：
 - 代码职责分离清晰，与论文结构一致
 - 但 Algorithm 1 中写的是"每 T 步做一次结构更新"，而代码实际是"每 T_es 步做一次"，且 T_es 是训练步数而非 ES
 轮数，与论文的 T（总 ES 轮数）概念不同
 - Algorithm 1 的 λ（候选数）在代码中由 generate_mutations() 动态生成（候选数取决于满足阈值的层数），不是固定的 λ

 结论：论文描述与代码有概念差异，需对齐
 ▎ Algorithm 1 中的 λ 改为"动态候选集 $\mathcal{N}(z_t)$"，说明候选数由满足阈值条件的层数决定，而非固定超参数。T
 改为"每 $T_{\mathrm{es}}$ 个优化步触发一次结构更新"。

 8. Power Iteration 高效初始化（算力解脱）

 论文描述（第207行）：
 ▎ "New components can be initialized near zero or from a gradient-informed rank-1 approximation."

 论文只提到"gradient-informed rank-1 approximation"，未说明如何高效计算（完整 SVD 需要 $O(d \cdot k)$ 内存）。

 代码实现（evo_rank_lora.py:230-269）：
 @staticmethod
 def _power_iteration_rank1(N, M, num_iters=5):
     """求 G ≈ N @ M 的主奇异三元组 (u₁, v₁, σ₁)
     复杂度 O(num_iters × (r·out + r·in))，不构造完整矩阵"""
     v = torch.randn(in_dim, ...)
     for _ in range(num_iters):
         u = N @ (M @ v); u /= u.norm()
         v = M.T @ (N.T @ u); v /= v.norm()
     return u, v, sigma

 分析：
 - 代码用 Power Iteration 替代完整 SVD，复杂度从 $O(d \cdot k \cdot \min(d,k))$ 降至 $O(r \cdot (d+k))$
 - 这是 Proposition 2（最优扩张方向）的高效实现，论文未说明实现细节
 - 5 次迭代在实践中足够收敛

 结论：代码实现更好，论文需补充
 ▎ 在 Practical Remarks 节补充：梯度主方向通过 Power Iteration（5次）高效计算，复杂度 $O(r(d+k))$，避免构造完整
 $d\times k$ 梯度矩阵。

 9. EMA 初始化防零偏差（防冷启动迟缓）

 论文描述（Algorithm 2, line 276-277）：
 Initialize EMA statistics û_ℓ ← 0 for all layers
 Initialize EMA statistics ŝ_{ℓ,i} ← 0 for all rank-1 components
 论文初始化为 0，使用标准 EMA 更新。

 代码实现（rank_evolution_controller.py:255-267）：
 if not self._is_initialized:
     self.ema_u[name] = curr_u      # 第一步直接赋值，不做 EMA
     self.ema_s[name][active] = curr_s[active]
 else:
     self.ema_u[name] = self.rho * self.ema_u[name] + (1-self.rho) * curr_u

 分析：
 - 论文初始化为 0，第一步 EMA 后 $\hat{u}^{(1)} = (1-\rho) \cdot u^{(1)}$（$\rho=0.9$ 时只有真实值的 10%）
 - 这会导致前几次结构更新时 EMA 值严重低估，阈值偏低，可能触发错误的扩张/剪枝
 - 代码第一步直接赋值（相当于 bias correction 的极端形式），消除冷启动偏差
 - 这是正确的工程决策

 结论：代码实现更好，论文需修正
 ▎ Algorithm 2 line 276-277 改为：第一次结构更新时直接赋值 $\hat{u}\ell \leftarrow u\ell^{(1)}$，后续使用标准
 EMA，避免零初始化导致的冷启动低估。

 max_reallocate_candidates: Optional[int] = 8  # 默认值

 分析：

 - 论文不应写成"默认就这样做"，而应写成"支持可选的候选限流"

 结论：论文需补充（保守写法），代码默认参数合理
 ▎ 在 Practical Remarks 节补充：为控制 ES 评估开销，Reallocation 候选数可通过 top-k cross 策略限制（默认上限
